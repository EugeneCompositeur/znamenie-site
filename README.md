# Театральный чат (Flask + SQLite)

Полностью рабочий веб-мессенджер для труппы до ~30 человек.

## Возможности
- Регистрация только по коду приглашения.
- Вход/выход, роли (`admin`, `member`).
- Чат с текстовыми сообщениями и тредами ответов.
- Реакции 👍 для сообщений и ответов (toggle, 1 пользователь = 1 реакция).
- Поиск по сообщениям и ответам.
- Календарь событий.
- Админ-панель: приглашения, пользователи, удаление сообщений.
- Автосоздание SQLite схемы при первом запуске.
- Базовая безопасность: хэширование паролей, CSRF-токен для POST, ограничение размера payload, secure cookie flags и security headers (CSP, X-Frame-Options).

## Структура
- `app.py` — Flask приложение и маршруты.
- `database.py` — инициализация SQLite и вспомогательные функции.
- `templates/` — HTML-шаблоны.
- `static/css/style.css` — стили (адаптивно).
- `static/js/chat.js` — логика чата (нативный JS).

## Локальный запуск
1. Установите Python 3.11+.
2. Создайте venv и установите зависимости:
   ```bash
   python -m venv .venv
   source .venv/bin/activate
   pip install -r requirements.txt
   ```
3. Задайте переменные (рекомендуется):
   ```bash
   export SECRET_KEY='change-me-super-secret'
   export INITIAL_INVITE_CODE='THEATRE-START'
   ```
4. Запустите приложение:
   ```bash
   python app.py
   ```
5. Откройте `http://localhost:5000`.

При первом запуске создается БД `theatre_chat.db` и стартовый активный код приглашения (`INITIAL_INVITE_CODE`).
Первый зарегистрированный пользователь автоматически получает роль `admin`.


## Как проверить локально без установки на сервер
1. Установите зависимости локально (в venv):
   ```bash
   pip install -r requirements.txt
   ```
2. Запустите автоматический self-check (без поднятия Gunicorn/Nginx):
   ```bash
   python scripts/selfcheck.py
   ```
3. Скрипт сам:
   - создает временную SQLite БД;
   - регистрирует первого пользователя (admin) по invite-коду;
   - отправляет сообщение в чат;
   - создает событие календаря;
   - проверяет доступ к админке.

Если self-check завершается строкой `Self-check passed ✅`, базовый функционал работает корректно.

## Продакшен (хостинг)

### Вариант A: Gunicorn + reverse proxy (Nginx)
1. Установите зависимости, как выше.
2. Запуск:
   ```bash
   gunicorn -w 2 -b 0.0.0.0:8000 app:app
   ```
3. Прокси через Nginx на `127.0.0.1:8000`.
4. Обязательно:
   - храните `SECRET_KEY` в переменных окружения;
   - ограничьте доступ к файлу SQLite и делайте бэкапы.

### Вариант B: PaaS (Render/Railway/другие)
- Build command: `pip install -r requirements.txt`
- Start command: `gunicorn -w 2 -b 0.0.0.0:$PORT app:app`
- Environment variables:
  - `SECRET_KEY`
  - `INITIAL_INVITE_CODE` (на время первого запуска)
  - `DB_PATH` (опционально, путь к sqlite)

## Безопасность и ограничения
- Пароли хранятся как хэши (`werkzeug.security`).
- CSRF-токен проверяется для всех POST-операций, токен сессии регенерируется при логине/регистрации.
- Пользовательский ввод экранируется в интерфейсе.
- Рекомендуется использовать HTTPS в продакшене и `SESSION_COOKIE_SECURE=1`.
- SQLite подходит для малой команды и умеренной нагрузки; для масштабирования замените на PostgreSQL.


## Известные ограничения
- Обновление чата реализовано polling'ом (каждые ~6 секунд), без WebSocket.
- Для команды до 30 человек SQLite подходит, но при росте нагрузки лучше перейти на PostgreSQL.
